# Product Requirements Document - git-manager

**Author:** Liuyang
**Date:** 2026-05-15

## Executive Summary

Git Manager 是一款面向开发者的桌面工具，核心解决"大量 Git 仓库的批量同步与管理"问题。产品起源于一个具体痛点：拥有数十上百个开源 GitHub 项目时，逐个进入目录执行 `git pull` 效率极低。从批量拉取工具逐步演化为工作目录管理器，当前已具备文件树浏览、文件操作、Git 集成（克隆、拉取、提交历史）和批量更新等能力。

**目标用户：** 需要同时维护大量本地 Git 仓库的开发者，尤其是参与多个开源项目或管理多个工作代码库的场景。

**产品定位：** 轻量级多仓库管理工具 — 不追求成为全功能 Git GUI 客户端，而是聚焦于多仓库场景下的批量操作效率。架构设计为未来扩展预留空间：它是一个**多工具集合桌面应用的起点**，Git 管理是第一个入驻的工具集，后续可按需扩展。

### 核心差异化

- **批量优先：** 区别于 SourceTree、GitKraken 等以单仓库为视角的 Git 客户端，以"多仓库全局视角"为设计原点 — 一键批量更新、全局状态总览
- **轻量高效：** 基于 Wails (Go + Vue 3) 构建，原生桌面性能，无 Electron 开销；安装即用，无需配置
- **可扩展的工具箱架构：** 愿景不局限于 Git 管理，而是演进为开发者日常工具集合平台

## 成功标准

### 用户成功

- **零摩擦访问：** 打开应用即可直达常用工作目录，无需逐级点击资源管理器
- **一键批量同步：** 所有仓库一键拉取更新，无需逐个进入目录执行 `git pull`
- **AI 工具链同步保障：** 常用 AI Skills / 工具仓库保持最新状态，消除"忘记更新"的焦虑

### 业务成功

- **个人效率工具定位：** 核心价值在于持续减少日常重复操作
- **AI 时代适配：** 确保 AI 工具快速迭代背景下，本地 Skills 和工具仓库始终最新

### 技术成功

- **启动即用：** 添加目录即可使用，无需复杂配置
- **批量操作可靠：** 并行拉取时正确处理成功/失败/超时，提供清晰进度反馈
- **轻量稳定：** 原生桌面性能，低内存占用，长时间运行无泄漏

### 可衡量指标

| 指标 | 目标 |
|------|------|
| 管理仓库数量 | 50+ 仓库无性能衰减 |
| 批量更新耗时 | 50 个仓库 2 分钟内完成（网络正常） |
| 启动时间 | 冷启动 < 3 秒 |
| 操作步骤减少 | 更新所有仓库：N 次手动操作 → 1 次点击 |

## 用户旅程

### 旅程 1：快速进入工作目录

**场景：** 上午开始工作，需要进入某个 AI Skills 仓库查看和编辑配置文件

1. 打开 Git Manager，左侧工作目录列表展示所有常用目录
2. 点击目标目录，文件树加载，右侧面板展示选中文件内容
3. 右键配置文件 → 在 VS Code 中打开并编辑
4. 编辑完成后，右键目录 → 更新仓库，一键 `git pull`

**价值：** "打开资源管理器 → 逐级点击 → 打开文件" 变为"一步直达"。

### 旅程 2：批量同步 AI Skills

**场景：** AI 工具迭代频繁，每天需确保所有本地 Skills 仓库是最新版本

1. 点击"批量更新"按钮
2. 应用递归扫描工作目录，识别 Git 仓库
3. 并行执行 `git pull`，实时展示每个仓库进度（成功/失败/跳过）
4. 查看汇总报告：哪些有新提交、哪些失败需关注
5. 对失败仓库查看错误详情，决定是否重试

**价值：** N 个仓库的 N 次 `git pull` 压缩为一次点击。

### 旅程 3：快速浏览和操作文件

**场景：** 需要查看某个仓库的配置文件内容

1. 在工作目录列表中找到目标仓库
2. 展开文件树，定位到目标文件
3. 点击文件，预览面板直接展示内容
4. 右键 → 复制文件名/路径，或新建文件

**价值：** 无需离开应用即可完成文件查看和基本操作。

### 旅程 4：克隆新项目

**场景：** 发现一个新的 GitHub 项目想要本地使用

1. 点击克隆按钮，输入 HTTPS/SSH 地址，选择本地路径
2. 克隆完成后，新仓库自动出现在文件树中，标记为 Git 仓库
3. 后续通过批量更新保持同步

**价值：** 从发现到可用只需一步。

### 旅程需求汇总

| 能力领域 | 关联旅程 | 核心要求 |
|----------|----------|----------|
| 工作目录管理 | 1, 2, 3, 4 | 添加/删除/默认目录、持久化配置 |
| 文件树浏览 | 1, 3 | 懒加载、Git 检测、隐藏文件夹显示 |
| 文件操作 | 3 | 创建、重命名、删除、预览、右键菜单、剪贴板 |
| Git 同步 | 1, 2, 4 | 单仓库拉取、批量并行拉取、进度反馈、错误处理 |
| 仓库克隆 | 4 | HTTPS/SSH 支持 |
| 外部工具集成 | 1 | VS Code 打开、资源管理器打开 |

## 产品范围与分阶段开发

### 策略

**MVP 策略：** 问题解决型 — 精准解决"多仓库批量 git pull"核心痛点，逐步扩展为轻量资源管理器。

**资源：** 单人开发，Go + Vue 3 全栈

### Phase 1 - MVP（已完成）

**核心旅程：** 旅程 1（快速进入工作目录）、旅程 2（批量同步）

- 工作目录管理（添加/删除/默认目录、JSON 持久化）
- 文件树浏览（懒加载、Git 仓库检测、隐藏文件夹显示）
- 文件操作（创建/重命名/删除/预览/右键菜单/剪贴板）
- Git 同步（仓库信息、克隆、单仓库拉取、批量并行拉取、提交历史）
- 三栏布局（目录列表 + 文件树 + 内容面板）

### Phase 2 - Growth（近期迭代）

**扩展旅程：** 旅程 3、旅程 4 增强

- 文件搜索功能
- 性能优化（大型仓库文件树加载、缓存策略）
- 外部工具集成（终端打开、默认程序打开）

### Phase 3 - Vision（远期规划）

- **多工具集合平台：** Git 管理作为第一个工具集，按需扩展其他开发者工具模块

### 风险缓解

| 风险类型 | 风险点 | 缓解策略 |
|----------|--------|----------|
| 技术 | 大型仓库文件树性能 | 懒加载 + 深度限制 + 缓存 |
| 技术 | Wails 框架限制 | 已投产验证，技术选型稳定 |
| 资源 | 单人开发精力有限 | 聚焦核心功能，按需迭代 |

## 桌面应用特定需求

### 平台支持

- **目标平台：** Windows 10/11（无跨平台计划）
- **框架：** Wails v2（Go + WebView2），原生桌面性能
- **分发：** `buildAndInstall.sh` 脚本构建安装，无自动更新机制

### 系统集成

- **已实现：** 文件资源管理器打开、VS Code 打开、系统剪贴板双向互通
- **扩展方向：** 按需添加终端打开、默认程序打开等
- **平台特性：** Windows 原生能力，无需跨平台抽象

### 离线能力

- **完全离线：** 文件树浏览、文件预览、文件操作、提交历史查看
- **需要网络：** git clone、git pull（失败仓库明确标记，不影响离线功能）

## 功能需求

### 工作目录管理

- FR1: 用户可以添加本地目录作为工作目录
- FR2: 用户可以从列表中移除工作目录
- FR3: 用户可以设置默认工作目录（应用启动时自动选中）
- FR4: 系统可以将工作目录列表持久化到本地配置文件
- FR5: 系统可以验证用户输入的路径是否有效

### 文件树浏览

- FR6: 用户可以展开工作目录查看其文件树结构
- FR7: 系统按需懒加载文件树子节点（展开时才加载）
- FR8: 系统可以自动检测目录是否为 Git 仓库并标识
- FR9: 用户可以查看隐藏文件夹（以 `.` 开头的目录，`.git` 除外）
- FR10: 用户可以全部展开或收起文件树
- FR11: 用户可以选中文件树节点查看其信息

### 文件操作

- FR12: 用户可以在指定文件夹下创建新文件
- FR13: 用户可以在指定文件夹下创建新子文件夹
- FR14: 用户可以重命名文件或文件夹
- FR15: 用户可以删除文件或文件夹
- FR16: 用户可以预览文件内容（文本文件直接渲染，二进制文件提示不可预览）
- FR17: 系统可以限制预览文件的大小（超出限制时提示）
- FR18: 用户可以通过右键菜单对文件/文件夹执行操作（操作列表根据节点类型区分）
- FR19: 用户可以复制文件名或文件完整路径
- FR20: 用户可以剪切、复制、粘贴文件和文件夹

### Git 仓库操作

- FR21: 用户可以查看 Git 仓库的基本信息（当前分支、远程地址、仓库状态）
- FR22: 用户可以克隆远程仓库到本地指定路径（支持 HTTPS 和 SSH）
- FR23: 用户可以对单个仓库执行 `git pull` 拉取更新
- FR24: 用户可以批量拉取多个仓库的更新（系统递归扫描工作目录下的 Git 仓库）
- FR25: 系统并行执行批量拉取，实时展示每个仓库的进度（成功/失败/跳过）
- FR26: 用户可以查看批量更新的汇总结果（成功数、失败数、失败原因）
- FR27: 用户可以分页查看仓库的提交历史

### 外部工具集成

- FR28: 用户可以在 VS Code 中打开文件或文件夹
- FR29: 用户可以在系统文件资源管理器中打开指定目录
- FR30: 用户可以用系统默认程序打开文件

### 应用框架

- FR31: 应用采用三栏布局（目录列表 + 文件树 + 内容面板）
- FR32: 应用可以显示当前版本号
- FR33: 应用的所有本地功能可在离线状态下使用（Git 网络操作除外）

## 非功能需求

### 性能

| 编号 | 要求 | 指标 |
|------|------|------|
| NFR1 | 应用冷启动时间 | < 3 秒 |
| NFR2 | 文件树单级目录加载 | < 1 秒（1000 个文件以内） |
| NFR3 | 文件预览渲染 | < 500 毫秒（1MB 以内文本文件） |
| NFR4 | 单仓库 git pull 超时 | 30 秒 |
| NFR5 | 批量更新并发控制 | 最多 5 个仓库并行拉取 |
| NFR6 | 批量更新 50 个仓库 | 2 分钟内完成（网络正常） |
| NFR7 | 内存占用 | 空闲 < 150MB，峰值 < 300MB |

### 安全

| 编号 | 要求 | 说明 |
|------|------|------|
| NFR8 | 路径安全 | 所有用户输入路径必须规范化处理，防止路径遍历攻击 |
| NFR9 | 文件操作确认 | 删除操作需用户二次确认，防止误删 |
