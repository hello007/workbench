## Context

当前项目是基于 Wails 框架的桌面应用，采用 Go 作为后端，Vue.js 3 + Element Plus 作为前端 UI 框架。项目已有基础的文件树浏览功能，能够识别 Git 仓库并提供基本的拉取操作。然而，在 UI 美观度和 Git 功能的深度展示方面还有提升空间。

现有架构中，前端通过 `wailsjs` 绑定调用后端 Go 方法，数据传输采用 JSON 格式。后端已经实现了 `GetFileTree`、`GetGitInfo` 等基础方法。项目使用 Element Plus 组件库，为 UI 美化提供了良好的组件基础。

## Goals / Non-Goals

**Goals:**
- 提升左侧文件树的视觉层次感和交互体验，使其更符合现代 UI 设计规范
- 为 Git 仓库提供完整的元数据信息展示，帮助用户快速了解仓库状态
- 实现提交历史的查看功能，支持用户追溯代码变更历史
- 保持良好的性能，即使在大型仓库中也能快速加载和渲染

**Non-Goals:**
- 不实现完整的 Git 操作功能（如 commit、push、branch 切换等）
- 不实现代码差异（diff）查看功能
- 不实现分支管理或合并冲突解决功能
- 不实现 Git 远程仓库的高级操作（如 fetch、rebase 等）

## Decisions

### 前端技术方案

**决定 1：使用 Element Plus 组件库增强 UI**
- **选择**：使用 Element Plus 的 `el-tree`、`el-card`、`el-timeline`、`el-tag`、`el-icon` 等组件
- **理由**：项目已集成 Element Plus，这些组件提供了丰富的定制选项和良好的视觉效果，能够快速实现美观的 UI
- **替代方案**：考虑过使用自定义 CSS 样式，但开发成本较高且难以保证一致性

**决定 2：文件树图标系统**
- **选择**：使用 Element Plus Icons（Folder、FolderOpened、Document、Link 等）配合自定义颜色方案
- **理由**：Element Plus Icons 与组件库风格统一，支持 SVG 渲染，具有良好的可扩展性
- **颜色方案**：
  - 文件夹：未展开 `#909399`，已展开 `#409EFF`
  - 文件：`#606266`
  - Git 仓库标识：`#67C23A`（绿色）
  - 悬停效果：背景色 `#F5F7FA`

**决定 3：Git 信息展示布局**
- **选择**：使用 `el-descriptions` 组件展示 Git 信息，采用卡片式布局
- **信息字段**：
  - 远程仓库地址（origin URL）
  - 当前分支
  - 最新提交 SHA
  - 最新提交作者
  - 最新提交时间
  - 最新提交消息
- **理由**：`el-descriptions` 提供了结构化的键值对展示方式，适合展示元数据信息

**决定 4：提交历史展示**
- **选择**：使用 `el-timeline` 组件展示提交历史，采用时间线布局
- **提交信息展示**：
  - 提交 SHA（前 8 位，可点击复制）
  - 提交消息（完整消息）
  - 作者名称和邮箱
  - 提交时间（相对时间格式，如"2 小时前"）
  - 变更文件数量
- **加载策略**：初始加载最近 20 条提交，支持滚动加载更多
- **理由**：时间线布局符合用户对历史记录的认知习惯，分页加载避免性能问题

### 后端技术方案

**决定 5：Git 操作库选择**
- **选择**：使用 `go-git` 库（如果当前未使用，需要新增依赖）
- **理由**：`go-git` 是纯 Go 实现的 Git 库，具有良好的跨平台支持，不需要系统安装 Git
- **替代方案**：使用 `os/exec` 调用系统 git 命令，但依赖系统 Git 且跨平台兼容性较差

**决定 6：新增 Go 方法**
- **GetGitRemoteURL(path string) (string, error)**：获取远程仓库地址
- **GetCommitHistory(path string, limit int, offset int) ([]Commit, error)**：获取提交历史
- **Commit 结构体**：
  ```go
  type Commit struct {
      SHA       string
      Message   string
      Author    string
      Email     string
      Timestamp int64
      Files     []string // 变更的文件路径列表
  }
  ```
- **理由**：将 Git 操作封装为独立方法，保持后端架构的清晰和可维护性

### 数据传输和性能优化

**决定 7：数据传输格式**
- **选择**：采用 JSON 格式传输数据，复用现有的 Wails 绑定机制
- **理由**：Wails 自动处理 Go 结构体和 JavaScript 对象之间的转换，减少手动序列化工作

**决定 8：性能优化策略**
- **选择**：
  - 提交历史采用分页加载（每页 20 条）
  - 文件树保持现有的懒加载机制
  - Git 信息按需加载（仅在点击 Git 仓库节点时加载）
- **理由**：避免一次性加载大量数据导致 UI 卡顿，特别是在大型仓库中

## Risks / Trade-offs

### 风险 1：大型仓库的性能问题
- **风险**：在大型仓库（如 Linux 内核、Chromium 等）中，加载完整的提交历史可能导致内存占用过高和 UI 响应缓慢
- **缓解措施**：
  - 实施严格的分页加载机制（每页 20 条）
  - 在后端限制单次返回的提交数量
  - 提供加载更多的按钮，避免自动无限加载

### 风险 2：跨平台兼容性
- **风险**：不同操作系统下的文件路径和 Git 配置可能存在差异
- **缓解措施**：
  - 使用 `filepath.Join()` 和 `filepath.Abs()` 处理路径，确保跨平台兼容
  - `go-git` 库已经处理了大部分跨平台问题
  - 在 Windows、Linux、macOS 上分别测试

### 风险 3：Go 方法调用的异步处理
- **风险**：Git 操作（特别是获取提交历史）可能耗时较长，如果同步调用会阻塞 UI
- **缓解措施**：
  - 使用 Vue 3 的 `async/await` 机制处理异步调用
  - 在加载过程中显示 loading 状态，提供良好的用户反馈
  - 考虑实现取消机制，允许用户中断长时间运行的操作

### 权衡 1：功能完整性 vs 开发时间
- **权衡**：完整实现 Git 历史查看需要支持很多高级功能（如 diff、分支切换、标签等），但这会大幅增加开发时间
- **决策**：优先实现核心的提交历史查看功能，其他高级功能留待后续迭代
- **理由**：快速交付用户最需要的功能，根据用户反馈决定后续开发优先级

### 权衡 2：UI 美观度 vs 性能
- **权衡**：丰富的动画和视觉效果可能影响渲染性能
- **决策**：在保证性能的前提下适度美化，避免过度复杂的动画
- **具体措施**：
  - 文件树展开/折叠使用 CSS transition（硬件加速）
  - 提交历史不使用虚拟滚动（Element Plus Timeline 不支持），但限制渲染数量
  - 使用 CSS `will-change` 属性优化动画性能

## Migration Plan

### 部署步骤

1. **依赖安装**
   - 如果需要新增 `go-git` 依赖：`go get github.com/go-git/go-git/v5`
   - 确认 Element Plus 已安装：`npm install element-plus`（应该已存在）

2. **后端开发**
   - 在 `app.go` 中新增 `GetGitRemoteURL` 和 `GetCommitHistory` 方法
   - 编写单元测试验证 Git 操作的正确性
   - 运行 `wails generate module` 生成 JavaScript 绑定

3. **前端开发**
   - 修改 `Home.vue` 中的文件树样式，添加图标和颜色方案
   - 新增 Git 信息展示组件（可以内联在 `Home.vue` 中或创建独立组件）
   - 新增提交历史组件（建议创建 `CommitHistory.vue` 组件）
   - 集成新增的 Go 方法调用

4. **测试**
   - 在不同大小的 Git 仓库中测试（小型、中型、大型）
   - 测试非 Git 仓库的行为（确保不影响现有功能）
   - 在不同操作系统上测试（Windows、Linux、macOS）

5. **构建和发布**
   - 运行 `wails build` 构建生产版本
   - 在目标平台上测试构建的安装包
   - 准备发布说明

### 回滚策略

- 如果新功能导致严重问题，可以通过 Git revert 快速回滚到上一个稳定版本
- 建议在新功能发布前打上 tag，便于快速回滚
- 由于改动主要集中在前端 UI 和新增 Go 方法，回滚不会影响现有功能和数据

## Open Questions

1. **提交历史的加载策略**
   - 问题：是否需要支持"加载全部"功能，还是严格限制为分页加载？
   - 影响：影响用户体验和性能
   - 待讨论：需要根据用户反馈决定

2. **Git 信息的缓存机制**
   - 问题：是否需要缓存 Git 信息和提交历史，避免重复加载？
   - 影响：提升性能但增加复杂度
   - 待讨论：如果性能测试表明有必要，可以实现简单的内存缓存

3. **提交历史的排序方式**
   - 问题：默认按时间倒序（最新的在前）还是正序（最老的在前）？
   - 影响：用户习惯
   - 待讨论：建议默认倒序，可以提供切换选项（如果用户需要）

4. **文件树的默认展开层级**
   - 问题：美化后的文件树是否需要默认展开到更深层级？
   - 影响：用户体验和初始加载性能
   - 待讨论：建议保持当前的懒加载机制，不改变默认行为
